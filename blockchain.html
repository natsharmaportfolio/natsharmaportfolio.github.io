Blockchain Program


How I coded it

I have 3 java classes: block, transaction, and blockchainInAction. The last class is essentially executing a hash of the previous block with the transactions all chained together. Each transaction occurs between two people; one person sending another person money or "data." The Block object class has 2 attributes: the hash of the previous block and the list of all the transactions in an ArrayList. The block then sets the nextHash() using the previousHash() with the hashCode() function.

The above diagram shows cipher block-chaining; a message or piece of data is split into blocks. Before encrypting, the first message m_0 is xor-ed with a random initialization vector. Subsequent block of ciphertext is xor-ed with next message block before encrypting. A new, random IV must be used with every encryption. Although this mode of block-chaining historically isn't secure, it's still an interesting baseline introduction to the concept of encryption.

I used my Algorithms and Security professors' publications and textbooks for reference when coding these programs: Applied Cryptography by Victor Shoup

Vigenere Cipher

How I coded it

The Vigenere cipher encrypts ciphertext using an algorithm: key i mod p. The example on the right of the diagram shows the structure of my code: my program lines up the user's message with the key and uses the vigenere matrix (on the left) to encrypt the message and return the ciphertext. The cipher can obviously be broken using frequency analysis, and in my next steps, I would like to create a program that can perform frequency analysis to crack a vigenere cipher.

Simplified Signal Protocol in Go

Implementing cryptographic logic for a secure messaging client, based on the Signal protocol, supported by Go Language

Security goal of this program is to achieve integrity and confidentiality of all messages transmitted, deniable message authentication, and forward secrecy. To make sure that my program's protocol is usable, I've implemented methods to handle errors and out-of-order messages.



Part 1: Triple Diffie-Hellman Handshake

A chat between A and B starts with a ​handshake w​here they exchange cryptographic key material and establish a shared session key. 3 DH exchanges are combined to authenticate both parties and produce a session. I implemented 3 methods: InitiateHandshake(), ReturnHandshake(), and FinalizeHandshake().



Part 2: Deriving forward-secure message keys with double ratchet

A and B need to derive symmetric keys for authenticated encryption. I implemented method DeriveKey() with the label CHAIN_LABEL. To achieve forward secrecy, after every message the sender/receiver ratchet the chain key to derive a new key.



Part 3: Adding resiliency with Diffie-Hellman ratchet

If attacker learns any of the chain key values, they could compute all values, so Signal adds 2nd layer of ratcheting. A updates root by calling CombineKeys().



Part 4: Handling out-of-order messages

Out-of-order messages include new DH ratchet value and need chain to be udpated. Each message contains LastUpdate value + sequence number.



Part 5: Handling Errors

EncodeData() function has been written in to encode data to binary, data which cannot be encrypted but needs integrity verification.



Part 6: Deleting key material

Cryptographic implementation must delete keys after use. Cannot wait for garbage collector to overwrite this value. So Zeroize() method is called on every key. EndSession() completely deletes all remining key material.
